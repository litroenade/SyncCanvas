## 一、项目定位与核心理念

### 1.1 项目定位

**SyncCanvas** 是一个基于 **Excalidraw** 的实时协作白板系统，通过自然语言控制已存在的画布元素，而非生成新的图表。

### 1.2 核心理念

> **"Control elements and not generate."**

**核心思想**:
- SyncCanvas 是一个尝试，用自然语言来获得更好的任务
- **不关注** LLM 将语言转换为 mermaid 或 tuple 的活动
- **目标是** 让 AI 能够控制已存在的画布，通过返回元素来实现
- 需要重构 Excalidraw 的保存方式，类似 **draw.io 的 mxCell** 结构
- 同时解决 draw.io **不支持多人实时协作**的问题

### 1.3 技术基础

| 基础组件 | 说明 |
|---------|------|
| **Excalidraw** | 底层 Canvas 渲染引擎 |
| **Yjs/CRDT** | 实时协作同步机制 |
| **类似 mxCell** | 元素存储数据结构（类似 draw.io） |
| **AI Agent** | 自然语言到元素操作的转换 |

### 1.4 核心价值主张

1. **控制而非生成**: AI 通过返回元素来控制画布，而非生成新图表
2. **实时协作**: 多人同时编辑，无冲突自动合并（draw.io 不具备）
3. **结构化存储**: 类似 mxCell 的元素存储，便于 AI 理解和操作
4. **自然语言交互**: 用自然语言描述任务，AI 自动操作画布元素

## 二、与竞品对比

### 2.1 SyncCanvas vs Excalidraw

| 特性 | Excalidraw | SyncCanvas |
|------|-----------|------------|
| **基础渲染** | ✅ 使用 Excalidraw | ✅ 基于 Excalidraw |
| **实时协作** | ⚠️ 需要额外配置 | ✅ 原生支持 CRDT |
| **AI 辅助** | ❌ 无 | ✅ 自然语言控制 |
| **元素存储** | JSON 数组 | ✅ 类似 mxCell 结构化 |
| **版本控制** | ❌ 无 | ✅ Git 式版本管理 |

### 2.2 SyncCanvas vs draw.io

| 特性 | draw.io | SyncCanvas |
|------|---------|------------|
| **元素存储** | ✅ mxCell 结构化 | ✅ 类似 mxCell 结构 |
| **实时协作** | ❌ **不支持多人同步** | ✅ **原生 CRDT 协作** |
| **AI 辅助** | ❌ 无 | ✅ 自然语言控制元素 |
| **渲染引擎** | mxGraph | Excalidraw |
| **数据格式** | XML (mxGraphModel) | JSON (Yjs CRDT) |
| **版本控制** | ❌ 无 | ✅ Git 式版本管理 |

**关键差异**:
- ✅ **draw.io 的优势**: mxCell 结构化存储，便于程序化操作
- ✅ **SyncCanvas 的优势**: 
  - 多人实时协作（draw.io 的痛点）
  - AI 自然语言控制
  - 基于现代化的 Excalidraw 渲染

### 2.3 SyncCanvas vs 其他 AI 绘图工具

| 特性 | 其他 AI 工具 | SyncCanvas |
|------|------------|------------|
| **工作方式** | 生成新图表 | ✅ **控制现有元素** |
| **输出格式** | Mermaid/图片 | ✅ **直接操作画布元素** |
| **实时协作** | ⚠️ 部分支持 | ✅ 原生支持 |
| **元素控制** | ❌ 无法精确控制 | ✅ 精确控制每个元素 |

## 四、核心功能规划

### 4.1 元素控制核心功能

#### 4.1.1 元素查询与检索

**功能描述**: AI 能够查询和检索画布上的现有元素

**实现方式**:
- `list_elements()`: 列出所有元素
- `get_element_by_id()`: 根据 ID 获取元素

- `find_elements_at_position()`: 根据位置查找

**工具示例**:
```python
@register_tool("list_elements")
async def list_elements(
    element_type: Optional[str] = None,
    context: AgentContext = None
) -> Dict[str, Any]:
    """列出画布上的元素，支持类型筛选"""
    # 返回元素列表，类似 mxCell 结构
    pass
```

#### 4.1.2 元素操作（CRUD）

**功能描述**: AI 能够对现有元素进行增删改查

**实现方式**:
- `update_element()`: 更新元素属性（位置、大小、样式、文本）
- `delete_elements()`: 删除元素
- `move_element()`: 移动元素
- `resize_element()`: 调整元素大小
- `update_element_style()`: 更新样式

**关键点**:
- ✅ **控制现有元素**，而非创建新元素
- ✅ 返回元素操作，直接更新 Yjs 文档
- ✅ 类似 mxCell 的精确控制

#### 4.1.3 元素关系管理

**功能描述**: 管理元素之间的连接和绑定关系

**实现方式**:
- `connect_elements()`: 连接两个元素（箭头/线条）
- `bind_elements()`: 绑定元素（如箭头绑定到节点）
- `group_elements()`: 分组元素
- `ungroup_elements()`: 取消分组

**类似 draw.io**:
- draw.io 的 mxCell 支持 `source` 和 `target` 属性
- SyncCanvas 的 `start_binding` 和 `end_binding` 实现类似功能

### 4.2 实时协作功能

#### 4.2.1 CRDT 同步

**功能描述**: 基于 Yjs/CRDT 的无冲突多用户编辑

**实现方式**:
- Yjs 文档存储元素（类似 mxCell 结构）
- pycrdt-websocket 服务器同步
- 增量更新机制

**优势**:
- ✅ **draw.io 不具备**的实时协作能力
- ✅ 自动冲突合并
- ✅ 低延迟同步 (< 100ms)

#### 4.2.2 房间管理

**功能描述**: 多房间隔离，每个房间独立的画布

**实现方式**:
- 房间级 Yjs 文档
- 房间级并发锁（防止多 AI Agent 冲突）
- 房间权限管理

### 4.3 AI Agent 系统

#### 4.3.1 自然语言理解

**功能描述**: 理解用户的自然语言指令，转换为元素操作

**工作流程**:
```
用户输入: "把支付节点移到右边，并连接到退款节点"
    ↓
AI Agent 理解:
  - 查找 "支付节点" 元素
  - 查找 "退款节点" 元素
  - 移动支付节点到右边
  - 创建连接（箭头）
    ↓
返回元素操作:
  - update_element(id="支付节点", x=新位置)
  - connect_elements(source="支付节点", target="退款节点")
```

#### 4.3.2 ReAct 循环

**功能描述**: 推理-行动-观察循环，逐步完成复杂任务

**流程**:
```
1. THINK - 分析用户意图和当前画布状态
2. ACT   - 执行元素操作（查询/更新/删除）
3. OBSERVE - 观察操作结果
4. 重复直到任务完成
```

#### 4.3.3 Agent 类型

| Agent | 职责 | 特点 |
|-------|------|------|
| **PlannerAgent** | 主协调者，理解意图，分发任务 | 任务规划 |
| **CanvaserAgent** | 专业绘图，处理复杂元素操作 | 元素控制专家 |

### 4.4 版本控制功能

#### 4.4.1 Git 式版本管理

**功能描述**: 类似 Git 的版本控制机制

| 概念 | Git 类比 | 说明 |
|------|---------|------|
| **Commit** | Commit | 完整状态快照（所有元素） |
| **Update** | Diff | 增量变更（元素操作） |
| **head_commit_id** | HEAD | 当前版本指针 |

**工作流程**:
1. 用户操作 → Yjs Update → 写入 Update 表
2. 用户离开 / 空闲超时 → 触发自动 Commit
3. 所有 Update 合并为 Commit
4. 更新 head_commit_id

### 4.5 数据持久化

#### 4.5.1 元素存储（类似 mxCell）

**存储结构**:
- Yjs 文档存储元素（运行时）
- SQLite 存储 Commit/Update（持久化）
- 元素以结构化方式存储（类似 mxCell）

**优势**:
- ✅ 便于 AI 理解和操作
- ✅ 支持精确的元素控制
- ✅ 支持元素关系查询

## 六、数据模型设计

### 6.1 元素模型（类似 mxCell）

```python
@dataclass
class CanvasElement:
    """画布元素 (类似 mxCell)
    
    统一的元素数据结构，支持所有 Excalidraw 元素类型。
    """
    id: str                    # 元素唯一标识
    type: ElementType          # 元素类型
    geometry: Geometry         # 几何信息
    style: Style              # 样式信息
    value: str                # 文本内容
    parent_id: Optional[str]  # 父元素 ID
    group_ids: List[str]      # 分组 ID 列表
    bound_elements: List[...] # 绑定的元素
    start_binding: Optional[Binding]  # 起始绑定（类似 mxCell source）
    end_binding: Optional[Binding]    # 结束绑定（类似 mxCell target）
    is_deleted: bool          # 是否已删除
    locked: bool              # 是否锁定
    version: int              # 版本号
```

### 6.2 数据库模型

```
Room
├── id, name, owner_id
├── head_commit_id → Commit
└── password_hash, is_public

Commit
├── id, room_id, parent_id
├── data (bytes) - 完整状态（所有元素）
├── hash (7字符)
└── author_name, message

Update
├── id, room_id
├── data (bytes) - 增量（元素操作）
└── timestamp

AgentRun
├── id, room_id, prompt
├── status, message
└── model, created_at

AgentAction
├── id, run_id
├── tool, arguments, result
└── created_at
```

## 结语

SyncCanvas 旨在通过自然语言控制画布元素，结合 Excalidraw 的渲染能力和 draw.io 的结构化存储优势，同时解决 draw.io 不支持多人实时协作的问题。

**核心价值**: **Control elements and not generate.**

让我们一起打造一个真正智能的协作白板系统！ 🚀
